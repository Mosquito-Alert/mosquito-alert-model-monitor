üìä Component Responsibilities

Tracked Projects

  • ‚úÖ Call monitoring via update_job_status.sh
  • ‚úÖ Continue execution regardless of monitoring success/failure
  • ‚úÖ Provide status updates (running, completed, failed)
  • ‚úÖ Include progress info (duration, percentage complete)

update_job_status.sh

  • ‚úÖ Write status files immediately (no waiting)
  • ‚úÖ Queue git operations via locked_git_sync.sh
  • ‚úÖ Never fail (always exit 0)
  • ‚úÖ Handle missing directories gracefully

locked_git_sync.sh

  • ‚úÖ Serialize git operations using file locks
  • ‚úÖ Handle lock timeouts (max 30 seconds)
  • ‚úÖ Clean up stale locks (> 5 minutes old)
  • ‚úÖ Retry failed pushes with rebase
  • ‚úÖ Preserve data locally if push fails

slurm_dashboard_sync.sh

  • ‚úÖ Periodic bulk sync (every 15-30 minutes)
  • ‚úÖ Use same locking mechanism as individual updates
  • ‚úÖ Minimal resource usage (512MB RAM, 1 CPU, 5 min max)
  • ‚úÖ Handle conflicts gracefully

Dashboard Pages

  • ‚úÖ Read status files for current state
  • ‚úÖ Read history files for trends
  • ‚úÖ Auto-refresh via GitHub Actions
  • ‚úÖ Handle missing data gracefully

üö® Race Condition Prevention

Problem Scenarios

  1. Multiple projects updating simultaneously
  2. Cron sync during project updates
  3. Git conflicts from concurrent pushes
  4. Stale locks from crashed processes

Solutions Implemented

  1. Immediate file writes - No waiting for git operations
  2. Git-only locking - Files can be written concurrently
  3. Lock timeouts - Prevent deadlocks
  4. Stale lock cleanup - Handle crashed processes
  5. Graceful failures - Preserve data even if sync fails

üìà Performance Characteristics

File Operations

  • Status file writes: < 1 second (no lock contention)
  • Multiple projects: Can write simultaneously
  • File size: ~1-2KB per status file

Git Operations

  • Lock acquisition: Usually instant, max 30 seconds
  • Commit/push time: 5-30 seconds depending on network
  • Conflict resolution: Automatic via rebase

Resource Usage

  • Individual updates: Minimal overhead
  • Cron sync: 512MB RAM, 1 CPU core, ~1-2 minutes
  • Dashboard build: Handled by GitHub Actions (free)
